# 中文互联网 ARG 表达手法与创作工具调研

更新时间：2026-07-20

## 0. 研究目的与口径

这份报告回答三个问题：

1. 中文互联网中被称为 ARG 的作品，实际使用了哪些叙事载体、谜题机制和运营方式？
2. 哪些只是页面的“皮肤”，哪些才真正改变玩家如何获得、验证和传播信息？
3. 如果要做一个开箱即用的 ARG 工具箱，应该抽象什么，而不是只堆一批仿制界面？

本次以[谜页集](https://miyeji.cn/)收录作品为主要中文样本入口，并补充銀河電燈、Eden's Code、中文 ARG 概念资料、ARG 设计论文和相关开源工具官方文档。详细逐项记录见[观察矩阵](OBSERVATION-MATRIX.csv)。

证据口径分为三档：

- **观察到**：作品介绍、公开页面或玩家反馈明确提及。
- **可推断**：从入口域名、公开交互或描述可以合理判断，但未得到作者技术说明。
- **建议支持**：不是在样本中确认的技术栈，而是工具箱的实现选择。

这一区分很重要：看到作品部署在 GitHub Pages，只能证明其公开入口使用该托管方式，不能反推出作者用了 React、Twine 或某个具体框架。

## 1. 中文互联网语境里的 ARG 到底是什么

经典定义强调：ARG 是把真实世界当作平台的互动网络叙事，通过跨媒体线索展开，玩家的想法或行动可能影响故事。中文资料通常还强调三点：跨媒体、与现实时间同步、过程不可逆；常用媒介包括网站、社交媒体、邮件、视频、电话、纸质材料和实地调查。[中文 ARG 词条](https://zh.moegirl.org.cn/ARG)同时列出 Puppet Master、Curtain、Rabbit Hole 和 TINAG 等核心术语。

但当前中文作品平台上的“ARG”是一个更宽的市场分类。大量作品实际上是：

- 单一网页承载；
- 可以随时开始、暂停和重玩；
- 由预写状态机响应，而不是幕后人员实时扮演 NPC；
- 用伪电脑、伪论坛、伪聊天、伪档案模拟跨媒体；
- 玩法更接近“拟态界面的调查叙事”或“网页解谜”。

这不是价值判断。它们通常更便宜、更稳定、更适合个人作者，也能制造很强的第一人称调查感。只是工具箱必须在概念上分开两种运行模式：

| 模式 | 本报告简称 | 时间 | 响应者 | 可重玩 | 典型形态 |
|---|---|---|---|---|---|
| 预编排的网页调查作品 | ARG-like Web | 玩家自己的时间 | 程序状态机 | 通常可以 | 假电脑、论坛、档案、聊天记录 |
| 现实中持续发生的跨媒体叙事 | Live ARG | 与现实同步 | PM、演员、服务与程序 | 通常不可完整复现 | 社媒、电话、邮件、直播、线下任务 |

理想的创作工具箱应共用同一套“真相—知识—证据—状态”内核，再选择 `static` 或 `live` 运行时。

## 2. 样本概况

截至调研时，[谜页集首页](https://miyeji.cn/)显示 160 部作品、80 位创作者，并将 ARG、网页解谜、互动叙事和现实互动解谜并列收录。样本有几个明显倾向：

- 免费、无需下载、无需登录、同时支持移动端和 PC，是主流低摩擦入口。
- GitHub Pages、Cloudflare Pages、Netlify 和独立域名都在使用；托管平台只是发布层，不等于创作工具。
- “失踪者留下的设备”“接手前任的电脑”“异常机构档案”“有问题的租房页面”是高频兔子洞。
- 主要动作不是战斗或移动，而是阅读、搜索、输入、比较、排序、解锁和决定是否回应。

代表样本：

- [《灵异论坛调查记录》](https://miyeji.cn/games/ling-yi-lun-tan-diao-cha-ji-lu)：模拟电脑与论坛，通过关键词搜索、翻帖和隐藏信息推进，包含分支与五个结局。
- [《查无此人》](https://miyeji.cn/games/cha-wu-ci-ren)：接手前任电脑，通讯录、工牌、邮箱、周报形成矛盾；三条调查线可并行，并设置假结局。
- [《邺山彼处》](https://miyeji.cn/games/ye-shan-bi-chu)：伪网站与现实互动结合；反馈明确提到手机扫码及外部“微信”环节，也暴露跨设备兼容和断链风险。
- [《我的秘密花园》](https://miyeji.cn/games/wo-de-mi-mi-hua-yuan)：玩家扮演自己，线索分布在文字、图片、链接和现实网络；反馈还表明听觉材料是推进的一部分。
- [《随时看房》](https://miyeji.cn/games/sui-shi-kan-fang)：伪租房平台、聊天记录、合同与电脑连接手机/ADB 的数码操作隐喻。
- [《少聊天室的女孩们》](https://miyeji.cn/games/game-1780534783225)：多聊天室对应多角色线，玩家可旁观或回应；不同回应产生不同结局碎片，多周目解锁真相线。
- [《回声：HK1939》](https://miyeji.cn/games/hk1939)：以大世界观、档案推理和复杂文字谜题为主。
- [《白色房间》](https://miyeji.cn/games/game-1779978379918)：以神秘邮件作为委托入口，强调连续谜题与数学要素。
- [銀河電燈作品页](https://ginga-light.top/works)：独立网站与微信公众号文章共同作为作品入口，说明公众号/文章流本身也可成为叙事载体。
- [Eden's Code](https://edenscode.cn/about.html)：高德地图、真实位置、NFC、WebAR、透明视频、拼图收集、行为任务和排行榜构成较完整的户外 ARG 技术组合。

## 3. 表达手法总谱

### 3.1 入口与现实框架

**兔子洞 / Trailhead**

第一条线索不只是“开始游戏”按钮。它可以是一封寄错的邮件、异常广告、失踪启事、招聘页面、租房信息、坏掉的机构网站、二维码、陌生账号的一条动态。好的兔子洞同时完成三件事：

1. 给出一个具体异常；
2. 赋予玩家介入的正当理由；
3. 暗示第一个可执行动作。

**玩家扮演自己**

中文样本中常见“你就是点开网页的网友/调查者”，而不是先创建虚构角色。这样能降低设定说明成本，并使公开检索、查看网页源代码、打开外部链接等现实动作自然进入玩法。

**伪网站 / 机构外壳**

新闻站、论坛、政府/企业档案、租房平台、个人博客、失物招领、内部系统都可作为可信外壳。关键不在像素级仿站，而在信息为什么会出现在这里、谁有权限修改、哪些痕迹本不该被玩家看见。

### 3.2 拟态界面载体

以下模块在网页 ARG-like 作品中最值得标准化：

| 模块 | 它擅长表达什么 | 可产生的玩法 |
|---|---|---|
| 桌面/手机壳 | “这是某人的设备”与统一入口 | 应用解锁、通知、时间变化、跨应用取证 |
| 邮件 | 正式关系、委托、附件、长时间跨度 | 发件人伪装、附件密码、邮件头、抄送关系 |
| 即时聊天 | 角色声音、延迟、删除与已读状态 | 搜联系人、选择回应、实时或伪实时 NPC |
| 论坛/贴吧 | 群体记忆、传言、旧帖沉积 | 关键词搜索、翻页、楼层暗号、账号关系 |
| 社交动态 | 公开人格和时间线 | 点赞关系、删帖、地理标签、跨账号矛盾 |
| 云盘/文件管理器 | 私密材料的层级与损坏感 | 文件夹路径、密码压缩包、版本历史 |
| 搜索引擎/档案库 | 玩家主动提出问题 | 精确关键词、别名、模糊结果、权限门 |
| 新闻/公告 | 公开叙事和机构口径 | 对照私人证词、日期矛盾、后续影响 |
| 文档/PDF/表格 | 制度化事实和视觉可信度 | 首字、批注、修订记录、表格异常 |
| 终端/日志 | 机器视角、不可见操作 | 命令、时间戳、错误码、隐藏目录 |
| 地图/GPS | 空间关系和现实位置 | 路线、坐标、地理围栏、现场任务 |
| 音频/语音信箱 | 口吻、环境与不可快速扫读的信息 | 倒放、频谱、背景声、电话号码 |
| 视频/监控/直播 | 不可靠画面与时间流 | 帧级线索、时间码、镜面、直播事件 |
| 相册/图像详情 | 现场痕迹和取景者视角 | EXIF、裁切、反射、图层、色彩通道 |
| 证据板/笔记 | 给玩家整理复杂关系的外部记忆 | 连线、标签、假设验证、提交推论 |

### 3.3 信息组织手法

**碎片化叙事**

同一事件由邮件、聊天、照片和日志分别描述。每份材料只承担一个角色视角，而不是把背景故事平均切成十段。

**知识不对称**

角色 A 知道事件但误解动机；角色 B 知道动机但没有见过事件；机构记录只有可计算事实；玩家通过交叉阅读获得任何单个角色都没有的认识。

**同一对象的多种口径**

账号名、身份证姓名、昵称、设备编号、论坛 ID 可以指向同一人，也可以故意指向多人。样本《查无此人》用“记录说不存在，但系统仍在运作”的冲突建立核心悬念。

**界面状态也是叙事**

已读、撤回、离线、最后修改时间、损坏、权限不足、正在输入、搜索零结果，都不是装饰。它们是角色行为留下的证据。

**环境叙事**

桌面壁纸、文件排序、应用使用频率、错别字、标点习惯和屏幕时间共同塑造设备主人。相比一篇“人物小传.txt”，这类分散痕迹更可信。

### 3.4 谜题原语

工具箱应支持可组合的原语，而不是只收录具体谜题答案：

1. **获取**：点击隐藏区域、翻页、滚动、检查附件、查看源代码。
2. **检索**：关键词、别名、编号、日期或地址。
3. **认证**：密码、PIN、恢复问题、角色权限。
4. **转换**：编码、替换、倒序、音频反转、图像通道。
5. **提取**：首字、固定位置、时间戳、文件元数据、频谱。
6. **排序**：时间线、路线、对话顺序、文件版本。
7. **关联**：跨媒介重复元素、账号关系、设备与地点匹配。
8. **矛盾验证**：公开记录与私人材料不能同时为真。
9. **选择/回应**：旁观、回复、公开、删除、交接。
10. **协作**：不同玩家获得不同片段，必须汇总。
11. **现实动作**：扫码、拨号、到达坐标、触碰 NFC、扫描图像。
12. **时间门**：到点推送、倒计时、错过即改变、现实日历事件。

最强的谜题通常不是“把摩斯码换成二进制”，而是让玩家使用剧情中已经建立的身份、习惯或关系完成推理。密码的答案应该改变玩家对故事的理解，而不只是打开下一扇门。

### 3.5 结构与节奏

**线性链**适合短作，但一个卡点会阻断全部体验。

**并行调查线**让玩家可以在人物、地点、机构三条线之间切换。《查无此人》的公开介绍明确使用三线并行；这是降低单点卡死又维持调查自由度的有效结构。

**钥匙—锁结构**要求从 A 获得细节，去 B 输入，再回 C 解释结果。它制造跨应用意义，但也最容易造成玩家忘记线索或不知道输入格式。

**假结局 / 多周目**可把第一次结案变成新证据。《查无此人》有假结局；《少聊天室的女孩们》用不同回应给出结局碎片和多周目真相线。

**分层提示**最好按“指出观察区域 → 指出关系 → 给出操作 → 给答案”逐级展开。提示也可以由角色口吻承载，但必须有脱离叙事包装的无障碍直达层。

## 4. “全部工具”应如何理解

无法仅凭页面穷举每位作者电脑上使用过的全部软件，也不应把网页依赖扫描等同于创作流程调查。因此这里给出三个层次的完整工具版图。

### 4.1 样本中明确披露或直接可见的工具/平台

| 类别 | 工具/平台 | 证据与用途 |
|---|---|---|
| 发布 | GitHub Pages | 多个入口为 `github.io`，适合纯前端静态作品 |
| 发布 | Cloudflare Pages | 多个入口为 `pages.dev`，适合静态/边缘 Web 部署 |
| 发布 | Netlify | 《回声：HK1939》入口为 `netlify.app` |
| 内容分发 | 微信公众号 | 銀河電燈作品直接以公众号文章作为入口 |
| 地图 | 高德地图 | Eden's Code 明确用于真实地点寻宝 |
| 物理交互 | NFC 信标 | Eden's Code 明确用于现场解锁 |
| WebAR | A-Frame + MindAR | Eden's Code 技术说明明确披露 |
| 跨端 | 微信/抖音小程序、移动/桌面浏览器 | Eden's Code 明确支持 |

### 4.2 工具箱推荐的内容生产工具

| 生产环节 | 推荐工具 | 用途 |
|---|---|---|
| 分支写作 | [Twine](https://twinery.org/) | 无代码起步、变量/条件、直接导出 HTML |
| 复杂叙事脚本 | [ink](https://www.inklestudios.com/ink/) | 适合频繁分支、汇流和可重写文本流程 |
| 流程图 | Mermaid / diagrams.net | 真相时间线、证据依赖和角色关系 |
| 图像 | Figma / Affinity / Photoshop / GIMP | 界面、证件、海报、截图与道具排版 |
| 音频 | Audacity / Reaper | 录音剪辑、噪声、声道、倒放与频谱谜题 |
| 视频 | DaVinci Resolve / Blender | 监控、新闻、Vlog、3D 与合成 |
| 媒体批处理 | [FFmpeg](https://www.ffmpeg.org/about.html) | 转码、切片、滤镜、音视频格式兼容 |
| 元数据 | [ExifTool](https://exiftool.org/exiftool_pod2.html) | 读写图片、音频、视频、PDF 元信息 |
| 二维码 | [node-qrcode](https://github.com/soldair/node-qrcode) | 浏览器/Node 生成 QR 入口或跨设备会话 |

### 4.3 工具箱推荐的运行时组件

| 能力 | 推荐实现 | 选择理由 |
|---|---|---|
| 状态与分支 | 自定义事件日志；复杂流程可用 [XState](https://github.com/statelyai/xstate) | 状态可视化、条件明确、便于检查不可达结局 |
| 地图 | [Leaflet](https://leafletjs.com/reference) | 开源、移动友好，也支持虚构平面地图 |
| 音频 | [Howler.js](https://howlerjs.com/) / Web Audio API | 跨浏览器播放、片段、循环、空间音频 |
| WebAR | [A-Frame](https://aframe.io/docs/) + [MindAR](https://hiukim.github.io/mind-ar-js-doc/) | HTML 风格声明式 3D 与图像追踪 |
| 存档 | localStorage / IndexedDB | 静态作品无需账号即可保存 |
| 协作状态 | Supabase / Firebase / Cloudflare Durable Objects | 多玩家共享进度、实时事件、运营控制 |
| 静态托管 | GitHub Pages / [Cloudflare Pages](https://developers.cloudflare.com/pages/configuration/git-integration/) | Git 推送自动构建，适合低成本作品 |
| 通知 | 邮件服务、短信/语音供应商、Web Push | Live ARG 的定时跨媒介触达 |

这些是工具箱的候选实现，不是对样本源代码的断言。

## 5. 从“文本”自动构建 ARG 的正确抽象

用户提出的三类输入——上帝视角文本、每个角色所能获得的信息、玩家所能获得的信息——已经接近正确底层，但仍需要编译为五张图：

1. **真相图**：世界里实际发生了什么，按因果和时间排序。
2. **知识图**：每个角色何时知道、相信、误解或隐瞒什么。
3. **证据图**：每个可见材料支持或反驳哪个命题，来源是否可靠。
4. **门槛图**：玩家需要何种操作、答案或状态才能获得下一证据。
5. **媒介图**：为什么这份信息存在于邮件而不是录音，它由谁创建、面向谁。

生成链条应是：

```text
上帝视角 + 角色知识 + 玩家初始知识
  → 事实原子与时间线
  → 知识差和待证明命题
  → 证据及可靠性
  → 谜题门槛与提示
  → 表现媒介匹配
  → 状态机、资源清单和可部署作品
```

表现工具不应由作者先随意勾选。系统应先推荐：

- 私密但正式、跨时间的信息 → 邮件；
- 即时关系和语气变化 → 聊天；
- 公共传言和多人视角 → 论坛；
- 机器记录且角色未必知情 → 日志；
- 空间因果 → 地图；
- 声音身份或环境证据 → 音频；
- 必须被角色保存的材料 → 云盘/档案；
- 需要公共传播后果 → 社交动态/新闻。

作者仍可覆盖推荐，但编辑器要警告“媒介没有叙事来源”或“同一事实被三份同样可靠的材料重复直说”。

## 6. 调研得到的产品原则

### 原则一：内容内核与界面皮肤彻底分离

一条证据应先是结构化对象，再由邮件、聊天或音频渲染器呈现。这样同一剧情可以换媒介而不重写全部逻辑。

### 原则二：自动生成的是第一版可玩编排，不是假装替代作者

模型可以提取事实、建议媒介、生成占位文本和检查漏洞；角色声音、伦理边界、谜题公平性与最终节奏仍需作者确认。

### 原则三：每条关键结论至少有可恢复路径

关键证据不应只藏在一个可能失效的外链、一次二维码扫描或一种设备专属操作里。应提供同世界观内的镜像、文字转写或替代路径。《邺山彼处》和《随时看房》的反馈都说明跨设备创意会直接产生兼容性卡死。

### 原则四：把“失败”写成世界响应

错误密码、零搜索结果和错过时间不该只是红字报错。它们可以增加日志、改变 NPC 态度、给出新关键词或打开旁路。

### 原则五：玩家需要外部记忆

超过 30 分钟、三个角色或五份材料后，应提供自动时间线、证据板、已发现关键词或可搜索笔记。否则难度来自工作记忆，而不是推理。

### 原则六：Live ARG 必须有运营控制台

完整 ARG 需要 PM 看见玩家卡点、延迟或撤回消息、切换备用方案、记录角色口径并冻结意外传播。单纯的页面生成器无法承担这些工作。

## 7. 安全、伦理与现实边界

ARG 会主动模糊虚构与现实，因此安全不是页脚声明就结束了。[ARG 伦理研究](https://www.researchgate.net/publication/320492534_Towards_an_ethics_of_alternate_reality_games)建议在设计、开发和运行全过程反复评估风险；混合现实游戏研究进一步将问题归纳为权利主张、照护义务、社会正义、隐私和控制等维度。[Niantic 玩家指南](https://nianticlabs.com/guidelines?hl=en)则给出现实地点游戏的直接底线：不闯入、不前往危险/无权进入区域、注意交通、尊重旁观者隐私。

工具箱应强制包含：

- 明确但不破坏氛围的虚构与退出说明；
- 内容警告、年龄建议、闪烁/声音/惊吓开关；
- 不冒充真实政府、医院、警方或具体个人；
- 不引导骚扰、开盒、试密码、闯入、尾随或危险驾驶；
- 虚构电话、邮件、域名和坐标的所有权检查；
- 现实地点必须有替代路径和白天/无障碍方案；
- 收集定位、邮箱、电话和聊天内容时最小化数据并明确同意；
- PM 紧急停服、统一澄清和角色失控预案；
- 外链失效、平台封禁、演员缺席时的降级方案。

TINAG 是审美策略，不是取消知情同意的许可证。

## 8. MVP 结论

第一版不要试图同时实现电话机器人、AR、NFC、直播演员和二十种界面。最有价值的闭环是：

1. 导入三类文本；
2. 自动提取时间线、角色知识差和 6–12 个证据命题；
3. 作者确认事实；
4. 系统推荐 3–5 种拟态界面；
5. 自动生成可通关状态机、分层提示和一份资源缺口清单；
6. 在浏览器预览并跑死路/泄底/不可达结局检查；
7. 导出纯前端站点，部署到静态托管。

首批渲染器建议只做：

- 桌面/手机壳；
- 邮件；
- 聊天；
- 云盘/文档；
- 搜索/档案；
- 图片与音频；
- 证据板；
- 结局与提示。

这八个模块已经能覆盖本次样本中大多数短篇中文网页 ARG 的核心体验。论坛、社交动态、地图、QR、实时 NPC、NFC 和 WebAR 作为后续插件。

更具体的数据结构、编译流程和模块 API 见[《ARG 工具箱规格》](ARG-TOOLBOX-SPEC.md)。
